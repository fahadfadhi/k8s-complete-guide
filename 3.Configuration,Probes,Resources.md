# 2. ConfigMaps (Non‑Secret Configuration)

A **ConfigMap** stores non-sensitive configuration data such as:

- Application settings  
- Environment variables  
- JSON/YAML config files  
- Feature flags  
- URLs  

ConfigMaps allow you to avoid hardcoding configuration inside container images.

---

## ConfigMap — Real-World YAML Example

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: web-config
data:
  APP_MODE: "production"
  LOG_LEVEL: "debug"

  config.json: |
    {
      "version": "1.0",
      "enabled": true,
      "max_connections": 100
    }
```

---

## Using ConfigMap as Environment Variables + File Mount

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: config-demo
spec:
  containers:
  - name: app
    image: nginx

    env:
    - name: MODE
      valueFrom:
        configMapKeyRef:
          name: web-config
          key: APP_MODE

    envFrom:
    - configMapRef:
        name: web-config

    volumeMounts:
    - name: cfg
      mountPath: /etc/config

  volumes:
  - name: cfg
    configMap:
      name: web-config
```

---

## ConfigMap Commands

```
kubectl create configmap web-config --from-literal=LOG_LEVEL=info
kubectl create configmap cfg --from-file=config.json
kubectl get cm
kubectl describe cm web-config
```

# 3. Secrets (Sensitive Configuration)

A **Secret** stores sensitive information such as:

- Passwords  
- API keys  
- TLS certificates  
- SSH private keys  
- Database credentials  

All values in a Secret must be **base64 encoded**, not encrypted.

---

## Secret — Real-World YAML Example

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  username: YWRtaW4=          # admin
  password: cGFzc3dvcmQ=      # password
```

---

## Using Secrets (ENV + Volume)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secret-demo
spec:
  containers:
  - name: mysql
    image: mysql:8

    env:
    - name: DB_USER
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: username

    envFrom:
    - secretRef:
        name: db-secret

    volumeMounts:
    - name: dbcreds
      mountPath: /etc/dbcreds

  volumes:
  - name: dbcreds
    secret:
      secretName: db-secret
```

---

## Important Notes About Secrets

- Base64 ≠ encryption  
- Should be encrypted at rest using KMS  
- Should avoid mounting Secrets in world-readable paths  
- Should rotate regularly

# 5. Downward API (Expose Pod Metadata to Containers)

The **Downward API** allows containers to access Pod information *without* calling the Kubernetes API.

It can expose:

- Pod name  
- Pod namespace  
- Pod IP  
- Node name  
- Pod labels  
- Resource requests/limits  

Used when applications need metadata about themselves.

---

## Downward API — Real-World YAML Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: downward-demo
  labels:
    app: demo
spec:
  containers:
  - name: app
    image: busybox
    command: ["sh", "-c", "sleep 3600"]

    env:
    - name: POD_NAME
      valueFrom:
        fieldRef:
          fieldPath: metadata.name

    - name: POD_IP
      valueFrom:
        fieldRef:
          fieldPath: status.podIP

    - name: NODE_NAME
      valueFrom:
        fieldRef:
          fieldPath: spec.nodeName
```

# 6. Probes (Health Checks)

Kubernetes uses three types of health probes:

---

## 6.1 Liveness Probe  
Checks if the container is **alive**.  
If it fails → Kubernetes restarts the container.

Used for:
- Stuck apps  
- Deadlocks  
- Long hangs  

---

## 6.2 Readiness Probe  
Checks if the container is **ready to serve traffic**.  
If it fails → Pod is removed from Service endpoints.

Used to:
- Wait for app startup
- Wait for DB connections
- Safely update apps

---

## 6.3 Startup Probe  
Used for **slow-starting applications**.  
Disables liveness/readiness until startup is complete.

Best for:
- Java apps (Spring Boot)
- Big frameworks

---

## Full Probe YAML Example (All Types)

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 10
  timeoutSeconds: 2
  failureThreshold: 3

readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  periodSeconds: 5

startupProbe:
  httpGet:
    path: /startup
    port: 8080
  failureThreshold: 30
  periodSeconds: 5
```

# 7. Resource Requests & Limits (CPU & Memory)

These control the Pod’s resource usage.

### **Requests**
- Minimum guaranteed resources
- Scheduler uses these to pick node

### **Limits**
- Maximum allowed resources
- If exceeded → throttling (CPU) or OOMKill (memory)

---

## Resource Requirements — YAML Example

```yaml
resources:
  requests:
    cpu: "200m"
    memory: "256Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

---

## Effects of Requests & Limits

### CPU:
- Requests → scheduler scoring  
- Limits → throttling only  

### Memory:
- Requests → scheduling  
- Limits → container killed if exceeded (OOMKill)

# 8. Lifecycle Hooks (postStart, preStop)

Lifecycle hooks allow containers to execute commands:

- Immediately after startup  
- Right before termination  

---

## Lifecycle Hook — Real Example

```yaml
lifecycle:
  postStart:
    exec:
      command: ["/bin/sh", "-c", "echo Container started"]

  preStop:
    exec:
      command: ["/bin/sh", "-c", "echo Shutting down && sleep 5"]
```

Used for:
- Graceful shutdown  
- Draining connections  
- Loading configuration

# 9. Init Containers (Extended)

Init containers run **before** the main application containers.  
They always run **in order**, one at a time.

Use cases:
- Waiting for database availability  
- Fetching configuration  
- Initializing folders  
- Performing migrations  
- Checking API availability  

Main container starts **only after all init containers complete successfully.**

---

## Init Container — Real-World YAML Example (Advanced)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: init-advanced-demo
spec:
  initContainers:
  - name: init-db
    image: busybox
    command: ["sh", "-c", "until nc -z mysql 3306; do echo waiting for db; sleep 2; done"]

  - name: init-config
    image: busybox
    command: ["sh", "-c", "echo Initializing config && cp /tmp/config/app.conf /etc/app/"]

  containers:
  - name: app
    image: nginx
    ports:
    - containerPort: 80
    volumeMounts:
    - name: config
      mountPath: /etc/app

  volumes:
  - name: config
    emptyDir: {}
```

# 10. Ephemeral Containers (Extended)

Ephemeral containers are NOT part of normal Pod specs.  
They are only used for **debugging running Pods**.

Key points:
- Added via *kubectl debug* command  
- Do not restart  
- Do not affect Pod restarts  
- Cannot be defined in YAML  
- Cannot be part of Deployment/ReplicaSet specs  

Used for:
- Debugging broken Pods  
- Inspecting filesystem  
- Running troubleshooting commands  

---

## Ephemeral Container — CLI Example

```
kubectl debug -it mypod --image=busybox --target=app
```

Where `app` is the container inside the Pod that you want to access.

# 11. Config Best Practices (Official Guidance + Real Experience)

### ✔ Do not put secrets in ConfigMaps  
Use Secrets for credentials, tokens, or sensitive data.

---

### ✔ Use envFrom for large config sets

```yaml
envFrom:
- configMapRef:
    name: app-config
```

---

### ✔ Mount big config files via volume  
Never put more than a few KB in env variables.

```yaml
volumeMounts:
- name: config
  mountPath: /etc/app
```

---

### ✔ Use init containers for setup tasks  
Don’t overload the main application container.

---

### ✔ Keep config out of container images  
Externalize everything so updates don’t require image rebuilds.

---

### ✔ Use downward API only when necessary  
Not for secrets — only for metadata.

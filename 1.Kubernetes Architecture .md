# Section 1 — Kubernetes Architecture (Full)

## 1. Kubernetes Cluster Overview

A Kubernetes cluster has two main parts:

### Control Plane (Brain of the cluster)
Responsible for:
- Scheduling
- Managing workloads
- Monitoring cluster state
- Handling failures
- API endpoint for all cluster communication

### Worker Nodes (Execution layer)
Responsible for:
- Running Pods
- Running containers
- Node-level networking
- Storage access

---

## 2. Control Plane Components

### 2.1 kube-apiserver
The kube-apiserver is the front-end of the Kubernetes control plane. All requests from kubectl, components, or applications pass through it.

Key functions:
- Validates API requests
- Processes REST operations
- Writes cluster state to etcd
- Authentication & Authorization enforcement
- Only component that communicates with etcd

---

### 2.2 etcd
etcd is a distributed, consistent key-value database used by Kubernetes to store all cluster state.

It stores:
- Pod definitions
- Services
- ConfigMaps, Secrets
- Nodes & health
- RBAC rules
- Deployments, ReplicaSets
- Network state

If etcd is lost, the cluster state is lost. Backups required.

---

### 2.3 kube-scheduler
Responsible for selecting which node a Pod should run on.

Scheduling steps:
1. Filter nodes (resource checks, taints)
2. Score nodes
3. Select best node
4. Bind Pod to node

---

### 2.4 kube-controller-manager
Runs core logic loops that keep desired state and actual state aligned.

Includes controllers:
- Node controller
- Deployment controller
- ReplicaSet controller
- Job controller
- EndpointSlice controller
- Namespace controller
- ServiceAccount controller

---

### 2.5 cloud-controller-manager
Used only in cloud environments (AWS, GCP, Azure).
Manages:
- Load balancers
- Nodes metadata
- Cloud routes
- Cloud volumes (EBS, PD, Disk)

---

## 3. Node Components

### 3.1 kubelet
The agent running on each node.

Responsibilities:
- Start containers
- Monitor containers
- Perform health checks
- Report node & pod status to API server
- Execute Pod specs as given by scheduler

---

### 3.2 kube-proxy
Implements Kubernetes Service networking.

Responsible for:
- Managing iptables/ipvs rules
- Load-balancing traffic to Pods
- Forwarding ClusterIP, NodePort, LoadBalancer traffic

---

### 3.3 Container Runtime
Runs containers inside Pods.

Supported runtimes:
- containerd (default)
- CRI-O
- Docker (legacy via CRI plugin)

---

## 4. Cluster Workflow Summary

Pod creation flow:
1. kubectl → API server (send YAML)
2. API server validates → stores in etcd
3. Scheduler finds node
4. API server updates etcd
5. Kubelet on chosen node starts containers
6. Kubelet reports back
7. API server updates etcd

Everything flows through API server.

## 5. Real-World Control Plane YAML (Expanded Example)

Below is a realistic, production-grade Deployment manifest
that demonstrates how the control plane components interact
when scheduling workloads. This Deployment includes:

- Rolling updates
- Probes (liveness, readiness, startup)
- Resource requests & limits
- Labels & selectors
- Node selection
- ServiceAccount usage

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  labels:
    app: web
spec:
  replicas: 3
  revisionHistoryLimit: 10

  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 1

  selector:
    matchLabels:
      app: web

  template:
    metadata:
      labels:
        app: web
    spec:
      serviceAccountName: web-sa

      nodeSelector:
        kubernetes.io/os: linux

      containers:
      - name: web
        image: nginx:1.27
        ports:
        - containerPort: 80

        resources:
          requests:
            cpu: "250m"
            memory: "256Mi"
          limits:
            cpu: "500m"
            memory: "512Mi"

        startupProbe:
          httpGet:
            path: /startup
            port: 80
          failureThreshold: 30
          periodSeconds: 5

        livenessProbe:
          httpGet:
            path: /health
            port: 80
          initialDelaySeconds: 10
          periodSeconds: 10

        readinessProbe:
          httpGet:
            path: /
            port: 80
          initialDelaySeconds: 5
          periodSeconds: 5
```

## 7. Control Plane Summary

- **API Server** = entry point for all communication  
- **etcd** = cluster database  
- **Scheduler** = picks correct node for Pods  
- **Controller Manager** = keeps system at desired state  
- **Cloud Controller** = manages cloud resources  
- **Kubelet** = node-level agent  
- **Kube-proxy** = service networking

# 9. How Kubernetes Handles Pod Lifecycle Internally

### Step-by-step explanation:

1. **User applies YAML**
   - Using `kubectl apply -f file.yaml`
   - Sent to API server

2. **API Server validates + stores in etcd**

3. **Scheduler notices unscheduled Pod**
   - Filters nodes
   - Scores them
   - Picks best node

4. **API Server updates Pod's assigned node in etcd**

5. **Kubelet on selected node**  
   - Sees assigned Pod
   - Pulls container image
   - Creates Pod sandbox
   - Starts containers

6. **Kube-proxy updates service routing rules**

7. **Pod running**
   - Kubelet sends health status to API server periodically

8. **If Pod fails**
   - Controller Manager recreates or replaces it (via Deployment/ReplicaSet)

# 10. Critical Notes for Architecture (Must Know)

### ✓ API Server NEVER talks directly to containers  
It ONLY talks to:
- etcd
- controller manager
- scheduler
- kubelet

### ✓ Scheduler NEVER starts containers  
Only decides the best node.

### ✓ Kubelet ONLY manages containers  
Does not schedule.

### ✓ etcd stores EVERYTHING  
It's the single point of truth.

### ✓ Most production outages happen because:
- etcd failure
- API server overload
- kubelet crash
- certificates expired

### ✓ All communication is over mTLS  
Every control-plane component uses certificates.

# Section 2 — Workloads (Full)

## 1. Introduction to Kubernetes Workloads

Kubernetes Workloads are the objects responsible for running applications in the cluster.
They define **how Pods are created, managed, scaled, updated, and scheduled**.

This section covers:
- Pods
- Multi-container Pods
- ReplicaSets
- Deployments
- DaemonSets
- StatefulSets
- Jobs
- CronJobs
- ReplicationController (legacy)
- Init Containers
- Ephemeral Containers
- Workload patterns
- Key commands

Everything in expanded Markdown for maximum readability.

# 2. Pods (Fundamental Workload)

A **Pod** is the smallest deployable unit in Kubernetes.  
It can run one or more containers that share the same:

- Network namespace (same IP, same ports)
- Storage volumes
- Process namespace (optional)

Pods are **ephemeral** — if a Pod dies, it is *not* recreated unless a controller manages it.

---

## Pod — Real-World YAML Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.27
    ports:
    - containerPort: 80

    resources:
      requests:
        cpu: "200m"
        memory: "256Mi"
      limits:
        cpu: "500m"
        memory: "512Mi"

    livenessProbe:
      httpGet:
        path: /health
        port: 80
      initialDelaySeconds: 10
      periodSeconds: 10

    readinessProbe:
      httpGet:
        path: /
        port: 80
      initialDelaySeconds: 5
      periodSeconds: 5

  volumes:
  - name: temp
    emptyDir: {}
```

---

## Pod Commands

```
kubectl get pods
kubectl describe pod nginx-pod
kubectl logs nginx-pod
kubectl exec -it nginx-pod -- bash
```

# 3. Multi-Container Pods

Pods can run multiple containers when they need to work together.
Common patterns:

- **Sidecar** (logging, monitoring)
- **Ambassador** (proxy)
- **Adapter** (transform data)

Containers share:
- Storage
- Network IP
- localhost communication

---

## Multi-Container Pod — Real Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: multi-container-pod
spec:
  containers:
  - name: app
    image: python:3.9
    command: ["python", "-m", "http.server", "8000"]

  - name: log-agent
    image: busybox
    command: ["sh", "-c", "tail -f /var/log/app.log"]

  volumes:
  - name: logs
    emptyDir: {}
```

# 4. ReplicaSets

A **ReplicaSet** ensures a specified number of identical Pods are always running.
If a Pod crashes or is deleted, the ReplicaSet replaces it automatically.

ReplicaSets are usually **not created directly** — Deployments manage them.
But understanding ReplicaSets is essential for workload orchestration.

---

## ReplicaSet — Real-World YAML Example

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: rs-web
  labels:
    app: web
spec:
  replicas: 3

  selector:
    matchLabels:
      app: web

  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: nginx
        image: nginx:latest
```

---

## ReplicaSet Commands

```
kubectl get rs
kubectl describe rs rs-web
kubectl delete rs rs-web
```

# 5. Deployments (MOST IMPORTANT WORKLOAD IN K8S)

A **Deployment** is a higher-level controller that manages ReplicaSets.

It provides:
- Rolling updates
- Rollbacks
- Version history
- Scaling
- Self-healing
- Zero downtime updates

Deployments are the **standard way** to run stateless applications.

---

## Deployment — Real-World YAML (Professional Example)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-deployment
  labels:
    app: web
spec:
  replicas: 3
  revisionHistoryLimit: 10

  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 1

  selector:
    matchLabels:
      app: web

  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: web
        image: nginx:1.25
        ports:
        - containerPort: 80

        resources:
          requests:
            cpu: "250m"
            memory: "300Mi"
          limits:
            cpu: "500m"
            memory: "600Mi"

        startupProbe:
          httpGet:
            path: /
            port: 80
          failureThreshold: 30
          periodSeconds: 5

        livenessProbe:
          httpGet:
            path: /health
            port: 80
          initialDelaySeconds: 10

        readinessProbe:
          httpGet:
            path: /
            port: 80
          initialDelaySeconds: 5
```

---

## Deployment Commands

```
kubectl apply -f web-deployment.yaml
kubectl get deployments
kubectl describe deployment web-deployment
kubectl rollout status deployment/web-deployment
kubectl rollout history deployment/web-deployment
kubectl rollout undo deployment/web-deployment
kubectl scale deployment/web-deployment --replicas=5
```

# 6. DaemonSets

A **DaemonSet** ensures that ONE Pod runs on **every node** (or on specific nodes).
Used for node-level tasks:

- Log shippers (Fluentd, Filebeat)
- Monitoring agents (Node Exporter, Datadog Agent)
- Network plugins (Calico, Cilium)
- Storage daemons

If a new node is added → DaemonSet automatically places a Pod there.  
If a node is removed → the Pod is removed.

---

## DaemonSet — Real-World YAML Example

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-monitor
  labels:
    app: monitor
spec:
  selector:
    matchLabels:
      app: monitor

  template:
    metadata:
      labels:
        app: monitor
    spec:
      containers:
      - name: node-exporter
        image: prom/node-exporter:latest
        ports:
        - containerPort: 9100
```

---

## DaemonSet Commands

```
kubectl get ds
kubectl describe ds node-monitor
kubectl delete ds node-monitor
```

# 7. StatefulSets (Ordered, Stateful Workloads)

A **StatefulSet** is designed for applications requiring:

- Stable network identity  
- Stable DNS  
- Stable storage  
- Ordered deployment & termination  
- Ordered rolling updates  

Used for **databases**:

- MySQL  
- PostgreSQL  
- MongoDB  
- Cassandra  
- Kafka  
- Zookeeper  

StatefulSets work together with:
- **PersistentVolumeClaim templates**
- **Headless Services**

---

## StatefulSet — Real-World YAML Example

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: "mysql"
  replicas: 3

  selector:
    matchLabels:
      app: mysql

  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql
        image: mysql:8
        ports:
        - containerPort: 3306

        env:
        - name: MYSQL_ROOT_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: password

        volumeMounts:
        - name: data
          mountPath: /var/lib/mysql

  volumeClaimTemplates:
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi
```

---

## StatefulSet Commands

```
kubectl get statefulsets
kubectl describe statefulset mysql
kubectl get pods -l app=mysql -o wide
```

# 8. Jobs (One-Time Workloads)

A **Job** runs a task **to completion**.

Used for:
- Batch processing  
- Database migrations  
- Backups  
- Data cleanup  
- Report generation  

Jobs guarantee execution until completion (unless failures exceed backoff limit).

---

## Job — Real-World YAML Example

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: data-migration
spec:
  backoffLimit: 3
  template:
    spec:
      restartPolicy: Never
      containers:
      - name: migrate
        image: busybox
        command: ["sh", "-c", "echo Running migration && sleep 5"]
```

---

## Job Commands

```
kubectl get jobs
kubectl describe job data-migration
kubectl logs job/data-migration
```

# 9. CronJobs (Scheduled Workloads)

A **CronJob** runs Jobs on a schedule, similar to Linux cron.

Used for:
- Nightly backups
- Log rotation
- Periodic cleanup
- Report generation
- Automated tasks

Format uses standard cron syntax:
```
* * * * *  
| | | | |
| | | | +---- day of week  
| | | +------ month  
| | +-------- day of month  
| +---------- hour  
+------------ minute
```

---

## CronJob — Real-World YAML Example

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: cleanup-job
spec:
  schedule: "*/5 * * * *"   # every 5 minutes

  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: Never

          containers:
          - name: cleanup
            image: busybox
            command: ["sh", "-c", "echo Running cleanup && rm -rf /tmp/*"]
```

---

## CronJob Commands

```
kubectl get cronjobs
kubectl describe cronjob cleanup-job
kubectl get jobs --watch
```

# 10. ReplicationController (Legacy)

**ReplicationController** is the older version of ReplicaSet.  
Still supported but **NOT recommended** for new workloads.

Replaced by:
- ReplicaSet
- Deployment

---

## ReplicationController — YAML Example

```yaml
apiVersion: v1
kind: ReplicationController
metadata:
  name: legacy-rc
spec:
  replicas: 2

  selector:
    app: legacy

  template:
    metadata:
      labels:
        app: legacy
    spec:
      containers:
      - name: nginx
        image: nginx
```

# 11. Init Containers (Run Before Main Container)

Init containers run **before** the application container starts.

Used for:
- Waiting for a service  
- Initializing directories  
- Running DB migrations  
- Downloading dependencies  
- Checking prerequisites  

Each init container must **complete successfully** before the next starts.

---

## Init Container — Real-World YAML Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: init-demo
spec:
  initContainers:
  - name: wait-for-db
    image: busybox
    command: ["sh", "-c", "until nc -z mysql 3306; do sleep 1; done"]

  containers:
  - name: app
    image: nginx
```

---

# 12. Ephemeral Containers (Debug Only)

Ephemeral containers are used ONLY for debugging **running Pods**.

They DO NOT:
- restart automatically  
- appear in Pod spec  
- run on Pod startup  

You cannot add them in YAML specs — only via CLI.

---

## Ephemeral Container — CLI Example

```
kubectl debug -it mypod --image=busybox
```

Creates a debug container **inside** the existing Pod.

# 13. Workload Summary

| Workload Type       | Purpose |
|---------------------|---------|
| **Pod**             | Smallest unit |
| **ReplicaSet**      | Ensures N Pods |
| **Deployment**      | Rolling updates + scaling |
| **DaemonSet**       | One Pod per node |
| **StatefulSet**     | Stateful apps with stable identity |
| **Job**             | Run task to completion |
| **CronJob**         | Run tasks on schedule |
| **Init Container**  | Run before main container |
| **Ephemeral Cont.** | Debug Pods |

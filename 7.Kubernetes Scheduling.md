# Section 7 — Kubernetes Scheduling (Full)

## 1. Introduction to Kubernetes Scheduling

Kubernetes scheduling decides **which node** each Pod should run on.  
The scheduler ensures:

- Efficient resource usage  
- High availability  
- Load balancing  
- Pod placement rules  
- Respecting taints, tolerations, affinities  
- Spreading workloads across zones/nodes  

This section includes:
- How the scheduler works  
- NodeSelector  
- Taints & Tolerations  
- Affinity & Anti-Affinity  
- Topology Spread Constraints  
- Priority & Preemption  
- Horizontal/Vertical Pod Autoscaling  
- Cluster Autoscaler  
- Debug commands  

---

# 2. How the Scheduler Works (Filter → Score → Bind)

The Kubernetes scheduler follows **three main steps**:

### **Step 1 — Filtering**
Remove nodes that cannot run the Pod:
- Not enough CPU/memory  
- NodeSelector mismatch  
- Taints not tolerated  
- Missing volumes  
- Node not Ready  

### **Step 2 — Scoring**
Remaining nodes are scored based on:
- Resource availability  
- Node load  
- Taint/toleration preferences  
- Topology spreading  
- Affinity preferences  

### **Step 3 — Binding**
Scheduler tells API server:
> “Place Pod X on Node Y”

Kubelet on that node then starts the Pod.

# 3. NodeSelector (Basic Scheduling Constraint)

NodeSelector is the simplest form of scheduling.

### 3.1 Label a Node

```
kubectl label node node1 disktype=ssd
```

### 3.2 Pod Using NodeSelector

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: node-selector-demo
spec:
  nodeSelector:
    disktype: ssd
  containers:
  - name: app
    image: nginx
```

Limitations:
- Only AND logic  
- No OR, NOT  
- No preference scoring  

Affinity solves these limitations.

# 4. Taints & Tolerations (Node Rejects Pods)

**Taints:** Applied on nodes  
**Tolerations:** Applied on pods  

Taints tell Kubernetes:
> “Do NOT schedule Pods here unless they tolerate this taint.”

### 4.1 Add a Taint

```
kubectl taint nodes node1 dedicated=db:NoSchedule
```

### 4.2 Pod With Matching Toleration

```yaml
tolerations:
- key: "dedicated"
  value: "db"
  effect: "NoSchedule"
```

### Taint Effects

| Effect | Meaning |
|--------|---------|
| **NoSchedule** | Pod will not schedule unless tolerated |
| **PreferNoSchedule** | Avoid if possible |
| **NoExecute** | Evicts running Pods without toleration |

# 5. Affinity & Anti-Affinity (Advanced Scheduling Rules)

Affinity lets you control **where Pods SHOULD run**,  
Anti-affinity controls **where Pods SHOULD NOT run**.

There are 3 types:

1. **nodeAffinity**  
2. **podAffinity**  
3. **podAntiAffinity**  

---

# 5.1 Node Affinity

Node affinity is like NodeSelector but more powerful:

- Supports AND, OR, NOT logic  
- Soft or hard rules  
- Match expressions  

### 5.1.1 Required (HARD rule)

Pod MUST run on a matching node:

```yaml
nodeAffinity:
  requiredDuringSchedulingIgnoredDuringExecution:
    nodeSelectorTerms:
    - matchExpressions:
      - key: disktype
        operator: In
        values: ["ssd"]
```

---

### 5.1.2 Preferred (SOFT rule)

Pod SHOULD run on a matching node:

```yaml
nodeAffinity:
  preferredDuringSchedulingIgnoredDuringExecution:
  - weight: 50
    preference:
      matchExpressions:
      - key: zone
        operator: In
        values: ["us-east-1a"]
```

---

# 5.2 Pod Affinity

Pod wants to run **near** another Pod.

Example: App needs to run close to cache.

```yaml
podAffinity:
  requiredDuringSchedulingIgnoredDuringExecution:
  - labelSelector:
      matchLabels:
        app: frontend
    topologyKey: "kubernetes.io/hostname"
```

Pod will run on the SAME node as `app=frontend`.

---

# 5.3 Pod Anti-Affinity

Pod wants to run **away** from matching Pods.

Used for:
- High availability  
- Spreading replicas  
- Avoiding resource contention  

```yaml
podAntiAffinity:
  requiredDuringSchedulingIgnoredDuringExecution:
  - labelSelector:
      matchLabels:
        app: redis
    topologyKey: "kubernetes.io/hostname"
```

This forces Redis replicas onto different nodes.

---

# 6. Topology Spread Constraints (Even Pod Distribution)

Ensures Pods are evenly spread across:
- Nodes
- Zones
- Regions

Useful for multi-zone HA.

### YAML Example:

```yaml
topologySpreadConstraints:
- maxSkew: 1
  topologyKey: "topology.kubernetes.io/zone"
  whenUnsatisfiable: DoNotSchedule
  labelSelector:
    matchLabels:
      app: web
```

This ensures Pods are evenly spread across zones.

# 7. Priority & Preemption

Pods can have **priority**, and when a high-priority Pod can't schedule,
Kubernetes can **evict lower-priority Pods** (preemption).

---

## 7.1 Define a PriorityClass

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: critical-apps
value: 1000000
globalDefault: false
description: "Critical system-level workloads"
```

---

## 7.2 Use PriorityClass in a Pod

```yaml
priorityClassName: critical-apps
```

High-priority Pods may preempt lower-priority Pods.

# 8. Horizontal Pod Autoscaler (HPA)

The **Horizontal Pod Autoscaler** automatically scales the number of Pods based on metrics.

Common metrics:
- CPU utilization
- Memory utilization
- Custom metrics (Prometheus Adapter)
- External metrics (cloud services)

### Requirements:
- Metrics Server must be installed  
- Resource requests must be defined  

---

## 8.1 HPA — Real-World YAML Example

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web-deployment

  minReplicas: 3
  maxReplicas: 10

  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```

---

## 8.2 HPA Commands

```
kubectl get hpa
kubectl describe hpa web-hpa
kubectl top pods
```

---

# 9. Vertical Pod Autoscaler (VPA)

VPA automatically adjusts:
- CPU requests/limits  
- Memory requests/limits  

Modes:
- **Off** — only generate recommendations  
- **Initial** — set resources on startup  
- **Auto** — update live (may restart Pods)

---

## VPA — Example YAML

```yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: web-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web-deployment
  updatePolicy:
    updateMode: Auto
```

---

# 10. Cluster Autoscaler (Increase/Decrease Nodes)

Cluster Autoscaler automatically adjusts the number of nodes based on:
- Pending Pods  
- Node underutilization  

Works on:
- AWS (ASG)  
- GCP (Instance Groups)  
- Azure VMSS  

Triggers:
- **Scale up:** Pod pending → no node fits  
- **Scale down:** Nodes empty → safe to remove  

Runs as a Deployment in the cluster.

# 11. Scheduling Debug Commands

### Describe scheduling decisions:
```
kubectl describe pod <pod>
```

### See scheduling events:
```
kubectl get events --sort-by=.lastTimestamp
```

### Check why a Pod is pending:
```
kubectl describe pod <pod> | grep -i failed
```

### View Metrics for scheduling:
```
kubectl top pods
kubectl top nodes
```

### Dry-run scheduling:
```
kubectl apply -f pod.yaml --dry-run=server
```

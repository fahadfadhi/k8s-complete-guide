# Section 4 — Kubernetes Networking (Full)

## 1. Introduction to Kubernetes Networking

Kubernetes networking is based on **simple but powerful rules**:

1. **Every Pod gets its own unique IP**  
2. **All Pods can communicate with all Pods without NAT**  
3. **Containers in the same Pod share the same network namespace**  
4. **Services provide stable virtual IPs to access Pods**

This networking model is implemented through:

- CNI plugins (Calico, Cilium, Flannel, Weave Net)
- kube-proxy (iptables/ipvs)
- Services (ClusterIP, NodePort, LoadBalancer, Headless)
- DNS (CoreDNS)
- Ingress Controllers
- Network Policies

This section explains ALL official networking concepts in detail.

---

# 2. Pod Networking (CNI)

Kubernetes does NOT handle Pod IP allocation itself.  
It uses **CNI (Container Network Interface) plugins)**.

CNI handles:

- Assigning Pod IPs
- Setting up routes
- Implementing network isolation
- Managing overlays (VXLAN, BPF)
- Enforcing NetworkPolicies

Popular CNIs:

- **Calico**
- **Cilium**
- **Flannel**
- **Weave Net**
- **Amazon VPC CNI**
- **Azure CNI**
- **GCP CNI**

---

## Check CNI plugin installed

```
kubectl get pods -n kube-system
```

---

# 3. kube-proxy (Service Routing Engine)

`kube-proxy` manages routing rules for Kubernetes Services.

Modes:

### iptables  
Stable, widely supported.

### ipvs  
Fast, scalable, ideal for large clusters.

---

## Check kube-proxy

```
kubectl get daemonset kube-proxy -n kube-system
```

---

# 4. Services (Stable Virtual IP Addresses)

A Service exposes a group of Pods using label selectors.

Types:

1. ClusterIP  
2. NodePort  
3. LoadBalancer  
4. ExternalName  
5. Headless  

---

# 5. ClusterIP Service (Default Internal Service)

Internal-only access.

Traffic:
```
Pod → ClusterIP → kube-proxy → Pod
```

---

## ClusterIP — Example

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-svc
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 8080
```

---

# 6. DNS (CoreDNS)

Handles DNS inside the cluster.

Example DNS name:
```
web-svc.default.svc.cluster.local
```

---

---

# 7. NodePort Service (Expose Service on Every Node IP)

A **NodePort** service exposes a port on *every* node in the cluster.

Traffic path:
```
Client → NodeIP:NodePort → kube-proxy → ClusterIP → Pods
```

Port range:
```
30000–32767
```

---

## NodePort — Real-World YAML Example

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-nodeport
spec:
  type: NodePort
  selector:
    app: web
  ports:
  - port: 80          # Service port
    targetPort: 8080  # Pod container port
    nodePort: 31080   # Exposed on each node
```

---

# 8. LoadBalancer Service (Cloud Load Balancer)

A **LoadBalancer** service provisions a cloud load balancer from:

- AWS (ELB/NLB)
- GCP Load Balancer
- Azure LB
- DigitalOcean LB

Traffic:
```
Internet → Cloud LB → NodePort → ClusterIP → Pods
```

---

## LoadBalancer — Real-World YAML

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-lb
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 8080
```

Kubernetes automatically assigns:
- External IP (cloud provider)
- Health checks
- Backend node mapping

---

# 9. Headless Service (Direct Pod Access)

Headless service does NOT allocate a ClusterIP.

```
spec:
  clusterIP: None
```

Used for:
- StatefulSets
- DNS discovery of individual Pod IPs
- Database clusters (MongoDB, Cassandra)

---

## Headless Service — YAML Example

```yaml
apiVersion: v1
kind: Service
metadata:
  name: mysql
spec:
  clusterIP: None
  selector:
    app: mysql
  ports:
  - port: 3306
```

DNS resolves as:
```
mysql-0.mysql.default.svc.cluster.local
mysql-1.mysql.default.svc.cluster.local
```

---

# 10. ExternalName Service

Maps a Kubernetes service name to an **external DNS name**.

Used to access:
- External databases
- External APIs

---

## ExternalName — YAML Example

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-db
spec:
  type: ExternalName
  externalName: db.example.com
```

# 11. Ingress (HTTP Load Balancing)

Ingress provides HTTP/HTTPS routing using an **Ingress Controller** such as:

- NGINX Ingress Controller
- AWS ALB Ingress Controller
- GKE Ingress
- Traefik
- Istio Gateway

Ingress provides:
- Host-based routing (app.example.com)
- Path-based routing (/api, /web)
- TLS termination
- Reverse proxy features

---

## Ingress — Real-World YAML Example

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web-ingress
spec:
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web-svc
            port:
              number: 80
```

Requires an ingress controller to work.

# 12. EndpointSlices (Modern Endpoint Management)

Kubernetes uses **EndpointSlices** instead of Endpoints for scalability.

Why?
- Endpoints object becomes too large in big clusters
- EndpointSlices split endpoint data into multiple lightweight objects
- Much more efficient and scalable

Each slice contains:
- Pod IPs
- Ports
- Readiness state
- Node info

---

## Check EndpointSlices

```
kubectl get endpointslices
```

# 13. Network Policies (Pod-Level Firewall)

NetworkPolicies control **which Pods can talk to which Pods**.

They enforce:
- Ingress rules (incoming traffic)
- Egress rules (outgoing traffic)

Important:
- They work only if CNI supports them (Calico, Cilium)
- Default is: all traffic allowed
- After applying NetworkPolicy → traffic is DENIED unless explicitly allowed

---

## NetworkPolicy — Real-World YAML Example (Allow Only Frontend → Backend)

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
spec:
  podSelector:
    matchLabels:
      app: backend

  policyTypes:
  - Ingress

  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
```

Meaning:
- backend Pods will only accept traffic from frontend Pods

# 14. Node Local DNS Cache (Optional Optimization)

Many cloud platforms install **NodeLocal DNSCache** to reduce cluster DNS load.

Benefits:
- Faster DNS lookups
- Better performance for large clusters
- Fewer CoreDNS bottlenecks

You will see a daemonset:
```
node-local-dns
```

# 15. DNS Deep Dive

Kubernetes automatically creates DNS entries for:

### Services
```
<service>.<namespace>.svc.cluster.local
```

### Pods (Headless Services only)
```
<podname>.<service>.<namespace>.svc.cluster.local
```

### SRV Records (for StatefulSets)
```
_service._protocol.<service>.<namespace>.svc.cluster.local
```

# 16. HostAliases (Override /etc/hosts Inside Pods)

You can override hostnames inside Pod without modifying image.

## Example:

```yaml
hostAliases:
- ip: "10.0.2.15"
  hostnames:
  - "internal.api"
```

# 17. NodeProxy (Internal Kubernetes Proxying)

Kubernetes includes a component called **kube-proxy**, but sometimes people say “node proxy” informally.

Node proxy (kube-proxy) responsibilities:
- Maintain iptables/ipvs rules
- Forward ClusterIP traffic
- Forward NodePort traffic
- Distribute traffic across Pod IPs
- Update rules when Pods are added/removed

It runs as a **DaemonSet on every node**.

# 18. Networking Debug Commands

### Check Pod IP & networking:
```
kubectl get pod -o wide
```

### Exec into Pod and test DNS:
```
kubectl exec -it <pod> -- nslookup web-svc
```

### Test service connectivity:
```
kubectl exec -it <pod> -- curl web-svc
```

### Show kube-proxy:
```
kubectl get ds kube-proxy -n kube-system
kubectl logs -n kube-system ds/kube-proxy
```

### Check NetworkPolicy:
```
kubectl describe networkpolicy
```

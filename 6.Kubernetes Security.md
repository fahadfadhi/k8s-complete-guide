# Section 6 — Kubernetes Security (Full)

## 1. Introduction to Kubernetes Security

Kubernetes security protects:
- **Cluster access** (Authentication & Authorization)
- **Pod-level access** (ServiceAccounts & RBAC)
- **Network security** (NetworkPolicies)
- **Secret management** (Secrets, encryption at rest)
- **API access control**
- **Pod security standards & isolation**

This section covers ALL major Kubernetes security concepts:
- Authentication (who are you?)
- Authorization (are you allowed?)
- Admission controllers
- ServiceAccounts
- RBAC
- Secrets security
- TLS & certificates
- Pod Security Standards
- Image security basics

Fully explained with real YAML + commands.

# 2. Authentication (Who is making the request?)

Kubernetes supports these authentication methods:

- **Client certificates** (kubeconfig)
- **ServiceAccounts (JWT tokens)**
- **Static tokens**
- **Webhook token authentication**
- **OpenID Connect (OIDC)**

User authentication is handled by the **API server**.

---

## kubeconfig Example (Used by kubectl)

A kubeconfig file contains:
- cluster info  
- user credentials  
- context  

Example entry:

```yaml
apiVersion: v1
clusters:
- name: mycluster
  cluster:
    server: https://1.2.3.4
    certificate-authority-data: <base64>

users:
- name: dev-user
  user:
    client-certificate-data: <base64>
    client-key-data: <base64>

contexts:
- name: dev-context
  context:
    cluster: mycluster
    user: dev-user
```

# 3. Authorization (Is the user allowed to perform the action?)

After authentication succeeds, Kubernetes checks **authorization**.

Kubernetes supports:
- **RBAC** (Role-Based Access Control)
- ABAC (deprecated)
- Node authorization
- Webhook authorization

RBAC is the standard.

# 4. ServiceAccounts (Identity for Pods)

Pods use **ServiceAccounts** for:
- API access
- Authentication with other cluster components
- Integrations with controllers

Default SA:
```
default
```

ServiceAccount is automatically mounted into Pods unless disabled.

---

## Creating ServiceAccount — YAML Example

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: web-sa
```

---

## Using ServiceAccount in Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: sa-demo
spec:
  serviceAccountName: web-sa
  containers:
  - name: app
    image: nginx
```

# 5. RBAC (Role-Based Access Control)

RBAC defines which users, groups, or ServiceAccounts can perform actions.

Primary RBAC objects:
- **Role** (namespace level)
- **ClusterRole** (cluster-wide)
- **RoleBinding**
- **ClusterRoleBinding**

---

## Example: Role (namespace access)

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: dev
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
```

---

## Example: RoleBinding (bind Role to user)

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: dev
subjects:
- kind: User
  name: dev-user
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

# 6. ClusterRole & ClusterRoleBinding (Cluster-wide Permissions)

A **ClusterRole** gives permissions across the entire cluster, not only one namespace.

Used for:
- Node access  
- PersistentVolumes  
- StorageClasses  
- Cluster-wide read/write  
- Admin roles  
- CSI drivers  
- Ingress controllers  

---

## ClusterRole — YAML Example

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: cluster-admin-view
rules:
- apiGroups: [""]
  resources: ["nodes", "pods", "namespaces"]
  verbs: ["get", "list", "watch"]
```

---

## ClusterRoleBinding — YAML Example

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: bind-admin-view
subjects:
- kind: User
  name: dev-user
roleRef:
  kind: ClusterRole
  name: cluster-admin-view
  apiGroup: rbac.authorization.k8s.io
```

# 7. Admission Controllers (API Request Filters)

Admission controllers sit **between authentication and persistence**.

Flow:
```
Auth → Admission → etcd
```

They can:
- Validate requests
- Mutate objects
- Enforce security policies
- Auto-inject sidecars
- Require labels or annotations

Two types:
- **Validating Admission Controllers**
- **Mutating Admission Controllers**

---

### Common Admission Controllers
- NamespaceLifecycle  
- LimitRanger  
- ServiceAccount  
- NodeRestriction  
- ResourceQuota  
- PodSecurity  
- MutatingAdmissionWebhook  
- ValidatingAdmissionWebhook  

---

# 8. Pod Security Standards (PSS)

PSS provides 3 security levels:

| Level | Meaning |
|--------|---------|
| **Privileged** | Anything allowed (for system daemons) |
| **Baseline** | Safe defaults, no dangerous settings |
| **Restricted** | Strongest isolation, zero privileges |

Applied using namespace labels:

```
pod-security.kubernetes.io/enforce: restricted
pod-security.kubernetes.io/audit: baseline
pod-security.kubernetes.io/warn: baseline
```

---

# 9. Pod SecurityContext (Controls Pod/User Privileges)

SecurityContext controls:
- Linux capabilities  
- File system permissions  
- User/group IDs  
- Privilege escalation  
- Read-only filesystem  
- SELinux/AppArmor  

---

## Pod SecurityContext — Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000

  containers:
  - name: app
    image: nginx
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
```

# 10. Secret Encryption at Rest

Kubernetes stores Secrets in **etcd**, base64‑encoded.  
To secure them, enable encryption-at-rest.

Encryption config lives in:
```
/etc/kubernetes/encryption-config.yaml
```

### Example:

```yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
- resources:
  - secrets
  providers:
  - aescbc:
      keys:
      - name: key1
        secret: <base64-encoded-key>
  - identity: {}
```

# 11. Certificates & TLS in Kubernetes

Kubernetes uses **mTLS (mutual TLS)** for nearly all internal communication.

Components using certificates:
- kube-apiserver  
- kubelet  
- controller-manager  
- scheduler  
- etcd  
- kubectl (client certificates)

Certificates usually found in:
```
/etc/kubernetes/pki
```

# 13. Image Security

Containers must use trusted images ONLY.

### Best practices:
✔ Use private registries  
✔ Enable image scanning (Trivy, Clair, Aqua)  
✔ Avoid running images as root  
✔ Pin versions (no `latest`)  
✔ Use distroless or slim images  
✔ Regularly patch images  

---

## Example Pod Using Non-Root Image

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-app
spec:
  securityContext:
    runAsUser: 1000
    runAsNonRoot: true
  containers:
  - name: app
    image: gcr.io/distroless/base
```

---

# 14. Restricting Container Capabilities

Drop Linux capabilities to minimize privileges.

```yaml
securityContext:
  capabilities:
    drop: ["ALL"]
```

---

# 15. Read-Only File System

Most secure workloads should not allow writing to root filesystem.

```yaml
securityContext:
  readOnlyRootFilesystem: true
```

# 16. API Access Restrictions for Pods

ServiceAccounts can be restricted using **RBAC**:

Example: Deny Pod from listing secrets:

```yaml
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: []
```

# 17. Kubernetes Audit Logging

Enabled at API server level.

Collects:
- who made request  
- what resource  
- what time  
- allowed/denied  

Useful for:
- Security monitoring  
- Investigations  
- Compliance  

Config file example:

```
--audit-log-path=/var/log/k8s-audit.log
--audit-policy-file=/etc/kubernetes/audit-policy.yaml
```

# 18. Security Debug Commands

### Check RBAC:
```
kubectl auth can-i get pods
kubectl auth can-i delete secrets --as dev-user
```

### View bound roles:
```
kubectl get rolebindings -A
kubectl get clusterrolebindings
```

### Inspect ServiceAccount token:
```
kubectl describe sa <name>
```

### Check API server certs:
```
openssl x509 -in /etc/kubernetes/pki/apiserver.crt -text
```

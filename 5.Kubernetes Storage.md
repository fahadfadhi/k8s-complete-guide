# Section 5 — Kubernetes Storage (Full)

## 1. Introduction to Kubernetes Storage

Kubernetes Storage is designed to give applications **persistent**, **portable**, and **reliable** data across Pods and nodes.

Key Concepts Covered in Section 5:
- Volumes
- Volume types (emptyDir, hostPath, configMap, secret, downwardAPI)
- PersistentVolumes (PV)
- PersistentVolumeClaims (PVC)
- StorageClasses
- Dynamic Provisioning
- CSI (Container Storage Interface)
- Projected volumes
- Ephemeral volumes
- Real-world YAML for all types
- Commands + deep concepts

This is one of the most important sections for stateful workloads.

---

# 2. Basic Volumes (non-Persistent)

Volumes provide storage that **lives for the lifetime of the Pod**.

Containers *inside the same Pod* share the same volume.

---

## 2.1 emptyDir (temporary storage)

Created when a Pod starts, **deleted when Pod is removed**.

Used for:
- Cache
- Temporary files
- Scratch space
- Shared storage for multi-container Pods

### emptyDir YAML Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: emptydir-demo
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: temp
      mountPath: /usr/share/nginx/html
  volumes:
  - name: temp
    emptyDir: {}
```

---

## 2.2 hostPath (mount host node’s filesystem)

Mounts a directory from **node’s OS** into the Pod.

Not recommended for production (node-specific).

### hostPath YAML Example

```yaml
volumes:
- name: host-storage
  hostPath:
    path: /data/app
    type: DirectoryOrCreate
```

---

## 2.3 configMap volume

Mounts ConfigMap data as files.

```yaml
volumes:
- name: config
  configMap:
    name: app-config
```

---

## 2.4 secret volume

Mounts Secrets as files.

```yaml
volumes:
- name: creds
  secret:
    secretName: db-secret
```

---

## 2.5 downwardAPI volume

Expose Pod metadata as files.

```yaml
volumes:
- name: meta
  downwardAPI:
    items:
    - path: "podname"
      fieldRef:
        fieldPath: metadata.name
```

# 3. Persistent Volumes (PV)

A **PersistentVolume (PV)** is a piece of storage in the cluster that has been provisioned by:
- Admin (static provisioning)
- StorageClass (dynamic provisioning)

PV lifecycle is independent of Pod lifecycle.

A PV contains:
- Capacity
- Access modes
- Reclaim policy
- Volume mode
- Storage backend details

---

## 3.1 PV — Real-World YAML Example

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-data
spec:
  capacity:
    storage: 10Gi
  accessModes:
  - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: manual

  hostPath:
    path: /mnt/data
```

---

# 4. PersistentVolumeClaim (PVC)

A **PersistentVolumeClaim (PVC)** is a request for storage by a Pod.

PVC requests:
- Size
- Access mode
- StorageClass

PVC binds to an available PV.

---

## 4.1 PVC — Real-World YAML Example

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pvc-data
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
  storageClassName: manual
```

---

# 5. Using PVC Inside a Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pvc-demo
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: storage
      mountPath: /var/data

  volumes:
  - name: storage
    persistentVolumeClaim:
      claimName: pvc-data
```

---

# 6. Access Modes Explained

| Access Mode | Meaning |
|-------------|---------|
| **ReadWriteOnce (RWO)** | Mounted by ONE node at a time |
| **ReadOnlyMany (ROX)** | Multiple nodes read-only |
| **ReadWriteMany (RWX)** | Multiple nodes read/write |
| **ReadWriteOncePod (RWOP)** | Single pod only |

Not all storage providers support all modes.

---

# 7. Volume Reclaim Policy

| Policy | Meaning |
|--------|---------|
| **Retain** | Keep data even if PVC deleted |
| **Delete** | Delete underlying storage |
| **Recycle** | Deprecated |

Most cloud volumes use **Delete**.

---

# 8. StorageClass (Dynamic Provisioning)

A **StorageClass** defines HOW Kubernetes provisions PVs dynamically.

It contains:
- Provisioner (AWS EBS, GCE PD, Azure Disk, CSI drivers)
- Parameters (SSD/HDD, IOPS, encryption)
- Reclaim policy
- Volume binding mode

---

## 8.1 StorageClass — Real Example (AWS EBS)

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3-storage
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  encrypted: "true"
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
```
---

# 9. Dynamic Provisioning Flow

1. Pod uses **PVC**  
2. PVC requests storage from a **StorageClass**  
3. StorageClass provisions a **PV** dynamically  
4. PV binds to PVC  
5. Pod mounts PVC  

This is the standard production pattern.

# 10. CSI (Container Storage Interface)

CSI is the modern standard for Kubernetes storage drivers.  
It allows third‑party storage vendors to integrate seamlessly.

Examples of CSI drivers:
- AWS EBS CSI
- Azure Disk CSI
- GCE PD CSI
- Ceph RBD CSI
- NetApp Trident
- Dell EMC PowerFlex
- Longhorn CSI
- OpenEBS

CSI enables:
- Dynamic provisioning
- Volume expansion
- Snapshots
- Clones
- Ephemeral CSI volumes

---

# 11. CSI Volume — Real-World YAML Example (AWS EBS)

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: ebs-claim
spec:
  accessModes:
  - ReadWriteOnce
  storageClassName: gp3-storage
  resources:
    requests:
      storage: 20Gi
```

Automatically provisions EBS volume using StorageClass:

```yaml
provisioner: ebs.csi.aws.com
```

# 12. Projected Volumes (Combine Multiple Sources)

Projected volumes let you combine multiple volume sources into **one directory**.

Sources that can be projected:
- Secrets  
- ConfigMaps  
- DownwardAPI  
- ServiceAccount tokens  

---

## Projected Volume — YAML Example

```yaml
volumes:
- name: app-config
  projected:
    sources:
    - configMap:
        name: web-config
    - secret:
        name: db-secret
    - downwardAPI:
        items:
        - path: "podname"
          fieldRef:
            fieldPath: metadata.name
```

---

# 13. CSI Ephemeral Volumes

Ephemeral CSI volumes exist **only for the Pod lifetime**.

Used for:
- Temporary high-performance disks  
- Scratch storage  
- Sidecar mounted volumes  

---

## CSI Ephemeral — YAML Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: ephemeral-csi-demo
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: scratch
      mountPath: /scratch
  volumes:
  - name: scratch
    csi:
      driver: local.csi.example.com
      volumeAttributes:
        size: "5Gi"
```

---

# 14. Volume Snapshots

Snapshots allow capturing **point-in-time** copies of volumes.

Useful for:
- Backups  
- Disaster recovery  
- Cloning environments  
- Rollback  

Requires CSI driver with snapshot support.

---

## Snapshot — YAML Example

```yaml
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: db-snapshot
spec:
  source:
    persistentVolumeClaimName: pvc-data
```

---

# 15. Volume Cloning

You can clone PVCs without taking snapshots first (CSI specific).

---

## PVC Clone — YAML Example

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: cloned-pvc
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
  dataSource:
    name: pvc-data
    kind: PersistentVolumeClaim
    apiGroup: ""
```

---

# 16. Expand PVC Storage (Resize Volumes)

StorageClasses must allow expansion:

```yaml
allowVolumeExpansion: true
```

Then modify PVC:

```
kubectl edit pvc pvc-data
```

Example change:
```
storage: 20Gi
```

# 17. Storage Debug Commands

### Check PV & PVC:
```
kubectl get pv
kubectl get pvc
kubectl describe pv <name>
kubectl describe pvc <name>
```

### Check which Pod is using PVC:
```
kubectl get pod -o wide
```

### Delete PVC but keep data:
```
kubectl patch pv <pv> -p '{"spec":{"persistentVolumeReclaimPolicy":"Retain"}}'
```

### Snapshot operations:
```
kubectl get volumesnapshots
kubectl get volumesnapshotclasses
```
